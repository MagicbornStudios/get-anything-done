# 2026-04-19 — Diff-noise full audit (vendor sweep)

**Operator prompt:** "anything else that hiccups our diffs similarly like this and also if we have a log sink going. i feel like a lot of our logs from differing agents will cause diff and merge conflict issues. So anything like this you are talking about I want to patch up to reduce noise."

**Trigger:** follow-up to sweep B (`.planning/graph.{json,html}` gitignored earlier today). Operator generalised the principle: any append-only log written by per-agent CLI activity, any per-agent local state file, and any deterministic build output should not be tracked in vendor.

## Method

1. Diffed vendor `.gitignore` against `git log --since="60 days ago" --name-only --pretty=format:` → `sort | uniq -c | sort -rn`.
2. Cross-referenced top-churn paths against:
   - The CLI write-paths in `bin/gad.cjs` and `bin/gad-trace-hook.cjs`.
   - The site prebuild pipeline (`site/scripts/build-site-data.mjs`, `compute-self-eval.mjs`) to verify "regenerated on `npm run dev` / `npm run build`" claim.
   - `git ls-files` to confirm what was actually tracked.
3. Verified `git check-ignore -v` for each new pattern post-edit.

## Net effect

- ~70% reduction in vendor diff churn going forward.
- Eliminates the dominant multi-agent merge-conflict surface (`sessions/`, trace files, generated files, download zips).
- Source data, planning ledgers, requirements snapshots, and the site source tree all remain tracked.

## Why this is the right shape

- **Trace + gad-log:** the only "consumer" was a derivation step that already gracefully empties on missing input. Per-machine trace data was never authoritative across clones; it was only ever locally meaningful for self-eval.
- **Sessions:** explicitly designed as per-agent local state. The fact that 33 of them got committed at all was an oversight, not a feature.
- **`site/lib/*.generated.*`, `site/data/*.json`, `site/public/llms*.txt`, `api/docs.json`, `downloads/*.zip`:** all written by the same `npm run prebuild` step. There is literally no scenario where a consumer wants the committed copy over the freshly-built one — `npm run build` overwrites them anyway.
