# v1.35.0 release is missing the npm tarball asset

Status: open · severity: high · captured 2026-04-18 by cursor-default-0026

## Symptom

`/gad:update` will fail for any user already on a runtime install pinned
to a version <1.35.0 the moment they try to update to v1.35.0.

The `run_update` step in `workflows/update.md` does:

```bash
gh release download "$LATEST_TAG" --repo "$REPO" --pattern 'get-anything-done-*.tgz' --dir "$TMP_DIR"
tar -xzf "$TMP_DIR"/get-anything-done-*.tgz -C "$TMP_DIR"
node "$TMP_DIR/package/bin/install.js" "$RUNTIME_FLAG" --global
```

`gh release download` exits non-zero because v1.35.0 has no asset
matching `get-anything-done-*.tgz`. The user sees a download error
instead of the update completing.

## Evidence

```text
$ gh release view v1.34.3 --repo MagicbornStudios/get-anything-done --json assets --jq '.assets[].name'
gad-v1.34.3-windows-x64.exe
get-anything-done-1.34.3.tgz
install-gad-windows.ps1
INSTALL.txt

$ gh release view v1.35.0 --repo MagicbornStudios/get-anything-done --json assets --jq '.assets[].name'
gad-v1.35.0-linux-x64
gad-v1.35.0-macos-arm64
gad-v1.35.0-windows-x64.exe
install-gad-windows.ps1
INSTALL.txt
```

The `get-anything-done-*.tgz` asset disappeared between v1.34.3 and
v1.35.0.

## Why this is a regression, not a deprecation

Decision **gad-274** is explicit: "**npm tarball stays for skill/hook/agent
install**; only SEA exe build deprecated." Phase-64 task `64-C` reaffirms
this: it deprecates `scripts/build-release.mjs` (the Node SEA path) and
keeps `publish:release` and `prepublishOnly` "unchanged for the npm
tarball/framework install chain." A test (`tests/release-deprecation.test.cjs`)
exists to lock this contract.

So the tarball is supposed to be in the release. It just isn't, in v1.35.0.

## Suspected cause

The release-binaries workflow (`.github/workflows/release-binaries.yml`)
gained the `GAD_RELEASE_BUILD=1` env wiring under task `64-E` for the Bun
build. The npm-tarball publish step is presumably a separate job (or a
separate `publish:release` invocation). Either:

1. The npm-tarball job was removed or no longer runs on tag push, or
2. The job runs but `gh release upload` for the `.tgz` was dropped, or
3. The job runs and uploads but the upload silently failed (no asset
   verification step).

A grep through `.github/workflows/` plus a manual `gh run view` on the
v1.35.0 release tag run will diagnose which.

## Fix candidates

| # | Fix | Notes |
|---|---|---|
| 1 | Restore the missing job/step in `release-binaries.yml` so `get-anything-done-${version}.tgz` is uploaded alongside the Bun binaries | Smallest change, preserves the current `update.md` flow |
| 2 | Re-publish v1.35.0 with the missing asset (`gh release upload v1.35.0 ./path/to/get-anything-done-1.35.0.tgz`) | One-shot recovery for the bad release; combine with #1 to prevent recurrence |
| 3 | Teach `update.md` to fall back to a Bun-binary install when the tarball is absent | Bigger change, only worth it if the project later commits to dropping the tarball entirely (would supersede gad-274) |

Recommend #1 + #2 together; do not pursue #3 without flipping gad-274.

## What landed 2026-04-18 (cursor-default-0026)

Fix #1 + a regression guard:

- `.github/workflows/release-binaries.yml` `publish` job now installs
  Node + deps and runs `npm pack --pack-destination dist/release-upload`,
  followed by a `Verify tarball asset present` step that exits non-zero
  if the tgz is missing. `npm pack` triggers `prepublishOnly` so
  `dist/gad.cjs` and `hooks/dist/` rebuild before packing.
- `scripts/publish-release.mjs` restored: `isReleaseArtifactName` accepts
  `${pkg.name}-${pkg.version}.tgz`; `ensureReleaseTarball()` now actually
  runs `npm pack` (idempotent); `getArtifacts()` calls it and asserts
  the tarball is present in the result set.
- `tests/release-deprecation.test.cjs` gains two regression cases:
  one for the script API, one for the workflow YAML contents.
