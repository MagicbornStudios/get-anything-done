# Handoff spec: Project market + scoped default views + eval philosophy

Paste everything below for the implementing agent.

## 1. Purpose

**Problem:** The playable archive on the home page is unbounded and already too large. Visitors still need a clear "things you can run / explore" surface without loading every historical run by default.

**Goal:** Ship a dedicated `/project-market` page that lists **all** eval projects the site already knows about, with sensible default filters (so the first paint is manageable) and featured placement for the two canonical product lines. Reduce or replace the full archive on the home page so the landing experience stays light.

**Principle:** Methodology is still work on a project. Anything you care about listing must be buildable (a real build or a defined artifact set that stands in for shipped output) and evaluable with artifacts available—not "vibes-only theory."

## 2. Naming & information architecture

### 2.1 Route

Primary route: `/project-market` (do not use `/evals` as the main public name for this catalog if the promise is projects/builds, not "eval docs only").

Rationale: Marketplace/catalog mental model; room for games, explainers, future domains.

### 2.2 Home page vs project market

- **Today:** Full Playable lives on home under `#play`; nav may use `/#play`.
- **Target:**
  - `/project-market` is the main place to browse all eval projects, filter, and open embeds (same core organism as today or extracted from it).
  - Home should not ship the full unbounded list by default. Options: teaser + link to project market, or minimal `#play` stub—implementation choice, but behavioral requirement is **size control** on the landing page.

### 2.3 Nav

Point Play / primary CTA to `/project-market` (keep `#play` only if you still anchor a small teaser; avoid implying the full archive lives only on home).

## 3. Catalog scope & defaults

### 3.1 All eval projects

The market lists **every** eval project exposed by the site's canonical data (same source that backs eval project pages / metadata today—e.g. generated eval index from `gad.json` / prebuild).

No hard allowlist that hides most projects.

### 3.2 Featured / canonical projects

**Escape the Dungeon (ETD)** and the **GAD explainer video** project are the two canonical lines that started the program.

They must be **featured** in the UI (pinned block, default sort to top, "Featured" badge, or equivalent—pick one pattern and document it in code).

They are not the only entries; everything else remains listable.

### 3.3 Default filtering (not exclusivity)

Apply defaults so first load isn't overwhelming. At minimum, honor:

- `ROUND_WINDOW_PER_PROJECT = 5`: show only runs whose round falls in the last five rounds **for that project** (see §3.4).
- Provide clear UI to widen (e.g. "Show all rounds", "All projects", clear filters) so power users can see full history.
- Featured projects may use the same defaults or slightly different initial filter presets—if so, document in one place (constants + comment).

### 3.4 Rounds are per project

"Last 5 rounds" means **per project** (eval id), not five rounds globally across the whole site.

Example: ETD (per slug or per grouped "ETD product"—see §3.5) has its own latest five; explainer has its own latest five; reader-workspace has its own five; etc.

### 3.5 ETD "family" presentation

The repo may have multiple project slugs for conditions (e.g. `escape-the-dungeon`, `escape-the-dungeon-bare`, `escape-the-dungeon-emergent`).

Implementer must choose (and document):
- **A)** One grouped "ETD" row/section with conditions inside, or
- **B)** Separate market rows per slug, each with its own five-round window.

Choice affects only presentation and grouping, not "whether other eval projects exist."

## 4. Methodology & evaluation model (product rules)

### 4.1 Methodology = build

Even a methodology track must be tied to a **buildable project** (something that produces a playable build or an explicit artifact bundle that substitutes for a shipped product).

No standalone "methodology" entry without a concrete eval project and evaluation path.

### 4.2 Artifacts

If you claim something is evaluated, artifacts must exist (or be generated by a defined pipeline): planning files, TRACE, preserved builds, CLI-parseable outputs, etc.

GAD self-eval is the reference: operational CLI + parsers + planning artifacts to show what is being done, pressure, loop compliance, etc.

### 4.3 With or without heavy tooling

- **With tooling:** CLI, skills, commands, hooks, snapshot, trace—full framework lane.
- **Without / minimal:** Still a project: typically AGENTS.md + requirements plus the pinned bootstrap skills (repo likely `find-skills` + `create-skill`—normalize names in implementation).

### 4.4 Typical testing matrix (Escape the Dungeon)

| Lane | Meaning |
|---|---|
| **GAD / framework** | Planning artifacts; optionally artifacts + CLI + skills + commands driving the framework. |
| **Bare / minimal** | Essentially AGENTS.md + requirements + two bootstrap skills only. |
| **Emergent** | Inherits skills authored in another eval; thereafter only those inherited skills plus the two bootstrap skills. |
| **Informal / injected** | Skills or stacks injected from outside (e.g. vendor pack, Phaser)—should become a new eval for that project, often templating from a prior eval. |

Cross-cutting: **You are always trying to build**; the matrix varies *how*, not *whether*.

### 4.5 Tech stack in requirements

Injecting tech stack (or otherwise changing the contract) → requirements change → typically a new round for that project (see §5).

## 5. Rounds vs requirements

- **Projects own rounds.**
- A new round corresponds to a new experimental epoch when the contract changes (e.g. stack injection, major requirement version).
- While a round has no new evals attached, you may update that round's requirements in place (clarifications) without minting a new round—until you branch with new evals or a new contract.
- Implementer: Map this to whatever the site already encodes as round / requirements version; if the model can't express "in-round edits," document the gap and implement filters from reliable fields first.

## 6. UX expectations (project market)

### 6.1 Core

- Grid or list of all eval projects with search/filter (reuse patterns from current Playable where possible: round, domain, status, hypothesis, text search).
- Default view: Featured block (ETD + explainer) + filtered run list per §3.3.
- Embeds: Same iframe / playable behavior as today for selected runs.
- Shareable state (optional but nice): URL query for project / round / selected run.

### 6.2 Home page

- No full unbounded archive as default.
- Prominent link to `/project-market`.

## 8. Acceptance criteria

1. `/project-market` exists and lists **all** eval projects from the canonical data source.
2. ETD (however grouped) and explainer are visibly **featured**, not the only rows.
3. Default view applies last 5 rounds per project (not global); users can expand to full history via filters/UI.
4. Home page does not default to the full unbounded archive; primary path points to project market.
5. Nav / key CTAs updated so `/project-market` is discoverable.
6. Build/typecheck pass; no dead `/#play` expectations without redirect or teaser.

## 9. Open implementation decisions (confirm in code)

## 10. Additional context from this session

### GAD CLI distinction (decision gad-128)

The GAD CLI is an **operational** CLI for the framework ecosystem. It is NOT a coding agent CLI. Claude Code CLI and Codex CLI are coding agent runtimes. GAD provides commands, agents, skills, and artifacts FOR those coding agent CLIs. This distinction must be clear on the site.

### Framework + catalog on same page

The /gad page should merge framework description and catalog. The skill bootstrap sets section should show hypotheses and what artifacts/tools are available per hypothesis condition (the testing matrix).

### Rubric on project detail (decision gad-131)

Move rubric display to project detail and eval run pages where it's contextual. /rubric can stay as methodology reference but is secondary.

### Methodology must build (decision gad-133)

No listing without a build or artifact set. GAD-on-GAD is the reference for methodology evaluation.
