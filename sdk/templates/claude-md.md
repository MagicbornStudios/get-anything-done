# CLAUDE.md Template

Template for project-root `CLAUDE.md` generated by `gad-tools generate-claude-md`.

Contains 7 marker-bounded sections. Each section is independently updatable.
The `generate-claude-md` subcommand manages 6 sections: project, stack, conventions, architecture, skills, and workflow enforcement.
The profile section is managed separately by the profile generator when that feature is in use.

---

## Section Templates

### Project Section
```
<!-- GAD:project-start source:PROJECT.md -->
## Project

{{project_content}}
<!-- GAD:project-end -->
```

**Fallback text:**
```
Project not yet initialized. Run `/gad:new-project` to set up planning artifacts.
```

### Skills Section
```
<!-- GAD:skills-start source:skills/ -->
## Project Skills

| Skill          | Description           | Path                      |
| -------------- | --------------------- | ------------------------- |
| {{skill_name}} | {{skill_description}} | `{{skill_path}}/SKILL.md` |
<!-- GAD:skills-end -->
```

**Fallback text:**
```
No project skills found. Add project-specific skills under `skills/` with a `SKILL.md` file for each skill.
```

**Discovery behavior:**
- Scans `skills/` for subdirectories containing `SKILL.md`
- Extracts `name` and `description` from YAML frontmatter when present
- Skips framework-installed skills prefixed with `gad-`
- Deduplicates by skill name

### Workflow Enforcement Section
```
<!-- GAD:workflow-start source:GAD defaults -->
## GAD Workflow Enforcement

Before using Edit, Write, or other file-changing tools, start work through a GAD entry point so planning artifacts and execution context stay in sync.

Use these entry points:
- `/gad:quick` for small fixes, doc updates, and ad-hoc tasks
- `/gad:debug` for investigation and bug fixing
- `/gad:execute-phase` for planned phase work

Do not make direct repo edits outside a GAD workflow unless the user explicitly asks to bypass it.
<!-- GAD:workflow-end -->
```

## Section Ordering

1. **Project** - Identity and purpose
2. **Stack** - Technology choices
3. **Conventions** - Code patterns and rules
4. **Architecture** - System structure
5. **Skills** - Project-specific skills available in `skills/`
6. **Workflow Enforcement** - Default GAD entry points for file-changing work
7. **Profile** - Developer interaction preferences

## Marker Format

- Start: `<!-- GAD:{name}-start source:{file} -->`
- End: `<!-- GAD:{name}-end -->`
- `source` enables targeted refreshes when upstream docs change
- Start markers should be unique and stable

## Fallback Behavior

When a source file is missing, fallback text should guide the runtime toward the correct next step.
Fallback text should not be a placeholder apology or internal implementation note.
